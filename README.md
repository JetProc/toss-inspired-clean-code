# toss-inspired-clean-code

토스 프론트엔드 클린 코드 스타일에서 아이디어를 얻어, 직접 관련 문제를 만들어 리액트 컴포넌트를 더 읽기 좋고 유지보수하기 쉬운 구조로 리팩토링하는 연습 저장소입니다.

공식 자료를 그대로 옮기기보다, 비슷한 문제 상황을 스스로 구성한 뒤 `before`와 `after`를 비교하면서 컴포넌트 분리, 조건 분기 정리, 추상화 레벨 조절을 연습합니다.


## 구조

```text
src/
  day{n}/
    before.{tsx,txt}
    after.{tsx,txt}
    ...
```

`day{n}` 디렉터리는 학습 순서대로 추가됩니다.

- `before.*`: 읽기 어렵거나 책임이 섞인 예제
- `after.*`: 같은 동작이나 구조를 더 명확하게 정리한 풀이
- 그 외 파일: 해당 예제 풀이에 필요한 훅, 유틸, 타입 등

## 다루는 주제

### 1부. 가독성 (Readability)

코드를 읽는 동료의 뇌내 연산 비용과 시선 이동을 최소화하는 단계입니다.

- [x] 1일차: 같이 실행되지 않는 코드 분리하기 (선언적 프로그래밍과 추상화)
- [x] 2일차: 구현 상세 파일 분리하기 (컴포넌트 진입점 비우기)
- [x] 3일차: 로직 종류에 따라 합쳐진 함수 쪼개기 (AuthGuard 조합 패턴과 파이버 트리 제어)
- [x] 4일차: 로직 종류에 따라 합쳐진 함수 쪼개기 2탄 (도메인 맥락에 따른 훅 분리와 DX 극대화)
- [x] 5일차: 복잡한 조건에 이름 붙이기 / 매직 넘버에 이름 붙이기 ('중복 제거 vs 가독성'의 딜레마)
- [x] 6일차: 시점 이동 줄이기 (연관된 코드 물리적 배치와 서브 컴포넌트 상태 격리)
- [x] 7일차: 삼항 연산자 단순하게 하기 (Early Return 패턴과 외곽 레이아웃 보존법)
- [x] 8일차: 왼쪽에서 오른쪽으로 읽히게 하기 (부정형 명세 정화 및 긍정 조건식 우선 배치)

### 2부. 예측 가능성 (Predictability)

누구나 코드를 보고 다음에 일어날 일과 반환값을 당연하게 예상하도록 만드는 단계입니다.

- [x] 9일차: 이름 겹치지 않게 관리하기 (Shadowing 방지와 직관적인 스코프 네이밍)
- [x] 10일차: 같은 종류의 함수는 반환 타입 통일하기 (일관된 API 응답 및 데이터 정형화)
- [x] 11일차: 숨은 로직 드러내기 (부수 효과 제거와 명시적 데이터 흐름)

### 3부. 응집도 (Cohesion)

수정해야 할 때 어디를 고쳐야 할지 명확하게 모아두는 단계입니다.

- [x] 12일차: 함께 수정되는 파일을 같은 디렉토리에 두기 (도메인 기반 폴더 구조 설계)
- [x] 13일차: 매직 넘버 없애기 (설정 데이터와 비즈니스 상수의 중심화)
- [x] 14일차: 폼의 응집도 생각하기 (수많은 Input 상태와 검증 로직 우아하게 한 곳에 모으기)

### 4부. 결합도 (Coupling)

하나의 변경이 다른 곳을 파괴하지 않도록 단단하게 격리하는 단계입니다.

- [x] 15일차: 책임을 하나씩 관리하기 (단일 책임 원칙 기반 커스텀 훅과 컴포넌트 분리)
- [ ] 16일차: 중복 코드 허용하기 (섣부른 공통화의 독과 의존성 결합 해제)
- [ ] 17일차: Props Drilling 지우기 (컴포넌트 주입 패턴과 Context API의 올바른 활용)

## 사용 방법

의존성을 설치한 뒤 타입 검사를 실행합니다.

```bash
npm install
npm run typecheck
```

새 예제를 추가할 때는 `src/day{n}` 디렉터리를 만들고, 가능하면 같은 동작이나 구조를 기준으로 `before`와 `after`를 함께 둡니다.
